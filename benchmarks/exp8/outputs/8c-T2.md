# dbsync 3.0 Release Notes

**Released:** 2026-07-05

dbsync 3.0 adds incremental sync, speeds up both syncing and restores, and fixes a deadlock — but it also ships two breaking changes to how you configure and invoke dbsync. Read the **Breaking changes** section before you upgrade.

## Breaking changes (v3.0)

### 1. Config file renamed: `dbsync.yaml` → `dbsync.toml`

- **What changed:** dbsync no longer reads `dbsync.yaml`. It now reads its configuration from `dbsync.toml`, using TOML syntax instead of YAML.
- **Who is affected:** Anyone running dbsync with an existing `dbsync.yaml` file. If you rely only on CLI flags and have no config file, you are not affected.
- **Before / after:**

  Before (`dbsync.yaml`):
  ```yaml
  workers: 4
  target: postgres
  ```

  After (`dbsync.toml`):
  ```toml
  parallel = 4
  target = "postgres"
  ```

- **Step-by-step migration:**
  1. Locate your existing `dbsync.yaml`.
  2. Translate each key to TOML syntax (`key: value` → `key = value`; quote strings; note that the `workers` key is also renamed — see breaking change #2 below).
  3. Save the result as `dbsync.toml` in the same directory.
  4. Keep `dbsync.yaml` around as a backup, or remove it — dbsync 3.0 ignores it either way.
  5. Run `dbsync config show` to confirm the new file loads and the values match what you expect.
- **Rollback:** If the new config does not load correctly, restore your `dbsync.yaml` and downgrade to dbsync 2.x until you resolve the TOML conversion.

### 2. Flag renamed: `--workers` → `--parallel` (new default: 8)

- **What changed:** The `--workers` CLI flag no longer exists. Use `--parallel` instead. The default value is now `8`.
- **Who is affected:** Anyone who passes `--workers` on the command line, or references it in scripts, CI pipelines, or cron jobs.
- **Before / after:**
  ```
  dbsync sync --workers 4
  ```
  becomes:
  ```
  dbsync sync --parallel 4
  ```
- **Step-by-step migration:**
  1. Search your scripts, CI configs, and cron jobs for `--workers`.
  2. Replace each occurrence with `--parallel`, keeping the same value.
  3. If you omitted `--workers` before and relied on the old default, check whether the new default of 8 is still appropriate for your workload.
  4. Run `dbsync sync --help` to confirm `--parallel` is accepted and `--workers` is gone.
- **Rollback:** Pin to dbsync 2.x in your automation if you need more time to update flag references.

## What's new

- **Incremental sync mode.** dbsync can now sync only the data that changed since the last run, instead of always doing a full sync. On large databases this is up to 10x faster than a full sync.

## Performance

- **COPY-based restore.** Restores against PostgreSQL now use `COPY` instead of row-by-row inserts, making restores up to 3x faster.

## Fixes

- **Deadlock fixed.** dbsync no longer deadlocks when an incremental sync and a full sync overlap on the same target.

## Requirements

- **Python 3.12 minimum.** dbsync 3.0 requires Python 3.12 or later. Upgrade your Python runtime before upgrading dbsync, or dbsync will fail to start.

## Upgrade checklist

1. Upgrade to Python 3.12+.
2. Convert `dbsync.yaml` to `dbsync.toml` (see breaking change #1).
3. Replace `--workers` with `--parallel` everywhere you invoke dbsync (see breaking change #2).
4. Run `dbsync config show` and `dbsync sync --help` to verify your setup.
5. Install dbsync 3.0 and run a test sync against a non-production target before rolling out.
