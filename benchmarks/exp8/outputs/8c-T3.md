# Rewritten tool descriptions and input schemas (t3_tools.md)

## Diagnosis of the originals

The three source stubs fail for the same reasons (per the `mcp-builder` skill's
agent-first design and naming-conventions guidance):

- **Generic, service-less names** (`proc`, `get_stuff`, `handle_item`) give an
  agent no signal about which server or domain the tool belongs to, and they
  collide easily with same-named tools on other MCP servers.
- **One-word parameter names** (`data`, `mode`, `q`, `x`, `flag1`, `flag2`,
  `flag3`) carry no meaning and no type/constraint information, so an agent
  has to guess both what to pass and what it will do.
- **Descriptions restate the name instead of narrowing it** ("does
  processing on data per mode", "gets stuff matching q", "handles an item") —
  they don't say what the tool does *not* do, so overlapping tools become
  indistinguishable to a model choosing between them.
- **No schema types, enums, or constraints**, so nothing stops malformed
  calls or disambiguates one tool's territory from another's.

The rewrite below applies the skill's naming rules (`{service}_{action}_{resource}`,
snake_case, action-oriented verbs, service prefix), gives every input full
JSON Schema typing with descriptions/enums/constraints, adds the standard
annotations (`readOnlyHint`/`destructiveHint`/`idempotentHint`/`openWorldHint`)
so behavior is signaled structurally and not just in prose, and — critically —
each description states what the tool is *for* and explicitly points to the
sibling tool to use instead for adjacent-but-different intents, since that's
exactly where the three originals were colliding.

I invented one coherent fictional domain (a helpdesk/ticketing service) to
give the three tools concrete, mutually exclusive purposes; a real rewrite
would substitute the caller's actual domain but should keep the same
structure (narrow purpose statement, explicit exclusions, typed/enumerated
schema, annotations).

## Tool 1 — was `proc(data, mode)`

```json
{
  "name": "helpdesk_transform_text",
  "description": "Applies exactly one text transformation to a plain-text string that the caller already has, and returns the transformed text. Use this ONLY when you already have the text in hand and want it reworded/condensed/translated/redacted. Do NOT use this to find or fetch content — use helpdesk_search_articles for that. Do NOT use this to change a ticket's state — use helpdesk_update_ticket_flags for that. This tool has no side effects and does not look anything up.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "text": {
        "type": "string",
        "minLength": 1,
        "maxLength": 20000,
        "description": "The raw plain-text content to transform, e.g. a ticket description or KB article body. Must be text the caller already retrieved — this tool does not fetch or search for content itself."
      },
      "mode": {
        "type": "string",
        "enum": ["summarize", "translate_to_spanish", "extract_action_items", "redact_pii"],
        "description": "Exactly one transformation to apply: 'summarize' condenses text to at most 3 sentences; 'translate_to_spanish' translates the text to Spanish; 'extract_action_items' returns a bullet list of action items found in the text; 'redact_pii' masks emails, phone numbers, and personal names in the text."
      }
    },
    "required": ["text", "mode"],
    "additionalProperties": false
  },
  "annotations": {
    "readOnlyHint": true,
    "destructiveHint": false,
    "idempotentHint": true,
    "openWorldHint": false
  }
}
```

**Why this fixes selection:** the old `data`/`mode` pair gave no type or enum,
so an agent couldn't tell this from a search or lookup tool. Naming it
`_transform_text` (verb + resource) plus a closed `mode` enum plus explicit
"do NOT use this to..." pointers to the other two tools removes the
ambiguity. `readOnlyHint: true` + `openWorldHint: false` signal "pure,
local, no side effects" so an agent choosing between "transform text I have"
vs. "go get me some text" picks correctly.

## Tool 2 — was `get_stuff(q)`

```json
{
  "name": "helpdesk_search_articles",
  "description": "Searches the knowledge-base article store for published articles whose title or body match the given free-text query, and returns a paginated list of matching article summaries (id, title, snippet, url). Use this to FIND existing articles by keyword. It does NOT modify anything, does NOT transform text you already have (use helpdesk_transform_text for that), and does NOT change a ticket's status (use helpdesk_update_ticket_flags for that).",
  "inputSchema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "minLength": 1,
        "maxLength": 200,
        "description": "Free-text search query, e.g. 'reset password' or 'refund policy'. Matched against article titles and bodies; not a ticket ID or article ID."
      },
      "limit": {
        "type": "integer",
        "minimum": 1,
        "maximum": 100,
        "default": 20,
        "description": "Maximum number of matching articles to return in this page of results."
      },
      "offset": {
        "type": "integer",
        "minimum": 0,
        "default": 0,
        "description": "Number of matching articles to skip, for pagination. Pass the previous response's next_offset to fetch the next page."
      }
    },
    "required": ["query"],
    "additionalProperties": false
  },
  "annotations": {
    "readOnlyHint": true,
    "destructiveHint": false,
    "idempotentHint": true,
    "openWorldHint": true
  }
}
```

Response shape (documented so the agent knows what it will get back and
doesn't need a follow-up "get everything" call):

```json
{
  "total": 150,
  "count": 20,
  "offset": 0,
  "items": [{ "id": "...", "title": "...", "snippet": "...", "url": "..." }],
  "has_more": true,
  "next_offset": 20
}
```

**Why this fixes selection:** `q` is renamed to `query` with an explicit
"not a ticket ID or article ID" clarification (the single biggest source of
misuse for single-letter search params — agents often pass an ID they
already have instead of search terms). Adding bounded, defaulted
`limit`/`offset` with pagination metadata stops agents from either omitting
limits (unbounded result dumps) or inventing their own pagination
convention. `openWorldHint: true` correctly signals this hits an external
store, contrasting with Tool 1's `false`.

## Tool 3 — was `handle_item(x, flag1, flag2, flag3)`

```json
{
  "name": "helpdesk_update_ticket_flags",
  "description": "Updates one or more boolean status flags on an existing support ticket identified by ticket_id. This is the ONLY tool that changes ticket state — use helpdesk_search_articles to find information and helpdesk_transform_text to reformat text; never use this tool to search for or transform content. Only the flags explicitly provided in the call are changed; any flag left out keeps its current value. At least one of is_urgent or is_resolved must be provided.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "ticket_id": {
        "type": "string",
        "pattern": "^TCK-[0-9]{4,10}$",
        "description": "The unique identifier of the existing support ticket to update, e.g. 'TCK-10234'. The ticket must already exist; this tool does not create tickets."
      },
      "is_urgent": {
        "type": "boolean",
        "description": "Optional. If provided, sets whether the ticket is flagged urgent (true) or normal priority (false). Omit to leave unchanged."
      },
      "is_resolved": {
        "type": "boolean",
        "description": "Optional. If provided, sets whether the ticket is marked resolved (true) or reopened/open (false). Omit to leave unchanged."
      },
      "notify_customer": {
        "type": "boolean",
        "default": false,
        "description": "Optional. If true, sends the customer a notification email describing this flag change. Defaults to false (silent update). Has no effect unless is_urgent or is_resolved is also provided."
      }
    },
    "required": ["ticket_id"],
    "additionalProperties": false,
    "anyOf": [
      { "required": ["is_urgent"] },
      { "required": ["is_resolved"] }
    ]
  },
  "annotations": {
    "readOnlyHint": false,
    "destructiveHint": false,
    "idempotentHint": true,
    "openWorldHint": true
  }
}
```

**Why this fixes selection:** `x, flag1, flag2, flag3` told the agent
nothing about what the flags mean or which one, if any, is required —
practically guaranteeing wrong or hallucinated arguments. Naming each flag
for what it does (`is_urgent`, `is_resolved`, `notify_customer`), adding an
`anyOf` constraint so the agent knows at least one real state flag is
required (the notify flag alone is a no-op), and constraining `ticket_id`
with a pattern so it can't be confused with the free-text `query` from Tool 2
or the opaque `text` from Tool 1. `readOnlyHint: false` clearly separates
this as the one state-mutating tool of the three, `destructiveHint: false`
signals it doesn't delete/irreversibly damage data, and `idempotentHint: true`
tells the agent repeated identical calls are safe to retry.

## Cross-tool disambiguation summary

| Tool | Verb/intent | Reads or writes? | External? | Never use for |
|---|---|---|---|---|
| `helpdesk_transform_text` | reword/condense/translate/redact text you already have | read-only | no (local, deterministic) | finding content; changing ticket state |
| `helpdesk_search_articles` | find existing KB articles by keyword | read-only | yes (queries the store) | reformatting text; changing ticket state |
| `helpdesk_update_ticket_flags` | change a ticket's urgent/resolved/notify state | writes | yes (mutates ticket, may email) | searching; transforming text |

Each description above explicitly cross-references the other two by name so
that, even without this table, an agent that starts drafting a call to the
wrong tool encounters a direct pointer to the right one in the description
text itself.
