# CCS Positioning Research (mid-2026)

Independent landscape scan of the "Compiled Composable Skills" (CCS) structure against prior art, gaps, and scope evidence. CCS elements referenced throughout: **minis** (6–12 focused self-contained markdown skill files), **INDEX** (knowledge-free <200-word router with "load when" hints), **00-core** (always-loaded cross-cutting mini), **BUNDLE** (compiled concatenation of minis, boundaries preserved, optional presets), and a **coverage/loading rule** (selective minis vs. whole bundle vs. subagent fan-out).

---

## 1. Structural prior art

**Anthropic Agent Skills (the closest analog).** The published spec implements exactly a three-level progressive-disclosure loader: Level 1 = YAML `name`+`description` metadata (~100 tokens/skill) always pre-loaded into the system prompt; Level 2 = the `SKILL.md` body (<5k tokens, target <500 lines) read via bash only when triggered; Level 3 = bundled reference files/scripts read or executed on demand with "no practical limit" and "no context penalty until accessed" ([overview](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview); [engineering blog](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills)). Best practices recommend "domain-specific organization" — a `SKILL.md` that navigates to `reference/finance.md`, `reference/sales.md`, etc. so unrelated context never loads ([best practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)). **Mapping to CCS:** a single Skill's `SKILL.md` = a per-skill INDEX+router; its `reference/*.md` files = minis; Level-1 metadata across many installed skills = a distributed INDEX. **What it LACKS vs CCS:** (a) no *compiled BUNDLE* — the model always reads files individually, there is no build step producing a single concatenated artifact or presets; (b) no explicit *coverage threshold* ("if <60% relevant load minis, else bundle"); (c) no formal *00-core* always-loaded cross-cutting mini within a skill (only global metadata); (d) references are deliberately kept "one level deep," which discourages CCS-style mini-to-mini composition. So Anthropic Skills = CCS minis + progressive disclosure, minus compilation, presets, and the coverage policy. CCS's compile step is the genuinely novel piece — everyone else keeps files separate at runtime.

**llms.txt / llms-full.txt (structural twin of INDEX + BUNDLE).** The spec defines `/llms.txt` = an H1 + blockquote summary + curated H2 link list (a knowledge-free navigation index) and `/llms-full.txt` = every doc concatenated into one file ([llmstxt.org](https://llmstxt.org/); [Mintlify](https://www.mintlify.com/docs/ai/llmstxt)). This is almost 1:1 with CCS's INDEX (router) + BUNDLE (compiled concatenation). **What it LACKS:** no per-topic "load when" trigger hints, no always-loaded core, no coverage rule, no minis-as-authored-units concept (it's generated *from* existing docs, not authored *as* modules). Note the adoption caveat below (§2).

**Cursor `.cursor/rules`.** Four rule types encode a coverage/loading policy directly in metadata: **Always** (`alwaysApply: true`, = 00-core, "use sparingly, 2–3 max"), **Auto-Attached** (glob-matched, file-path triggers), **Agent-Requested** (description-only, model decides relevance — = INDEX "load when" hints), and **Manual** (`@rule` invocation) ([Cursor docs](https://cursor.com/docs/rules)). **Mapping:** rules = minis; the always/auto/agent-requested taxonomy ≈ CCS core + selective loading. **LACKS:** no compiled bundle, no index-of-minis file, no preset concept; triggering is per-rule not corpus-level.

**GitHub Copilot `.github/instructions/*.instructions.md`.** Multiple modular instruction files, each with `applyTo:` glob frontmatter and an `excludeAgent` field ([GitHub docs](https://docs.github.com/en/copilot/how-tos/copilot-on-github/customize-copilot/add-custom-instructions/add-repository-instructions); [VS Code](https://code.visualstudio.com/docs/agent-customization/custom-instructions)). = CCS minis with glob-based auto-attach; a repo-root `copilot-instructions.md` acts like an always-on core. **LACKS:** index, bundle, presets, coverage rule.

**AGENTS.md.** Open standard (OpenAI/Google/Cursor/Factory, Aug 2025; donated to the Linux Foundation's Agentic AI Foundation Dec 2025; 60k+ repos, 20+ tools) ([PRPM](https://prpm.dev/blog/agents-md-deep-dive); [ASDLC](https://asdlc.io/practices/agents-md-spec/)). It is the *monolithic* opposite of CCS: one always-loaded file, no modular decomposition, index, or disclosure — the strong incumbent CCS differentiates against.

**MCP prompts & resources.** MCP servers expose reusable, parameterized prompt templates and embeddable resources, "versioned and updated centrally" ([spec](https://modelcontextprotocol.io/specification/2025-06-18/server/prompts)). = distributable minis + a listing endpoint (index-like) with versioning CCS lacks. **LACKS:** compiled bundle, core, coverage policy; prompts are client-surfaced, not model-autoloaded.

**Claude Code plugins/marketplaces.** A plugin bundles skills/MCP/commands; a marketplace is a git repo with `marketplace.json` giving centralized discovery, version pinning, auto-update ([plugin marketplaces](https://code.claude.com/docs/en/plugin-marketplaces)). This is the *distribution/registry* layer CCS has no equivalent of.

**DSPy.** Composable **modules** (each wrapping a technique/signature) + a **compiler/optimizer** that assembles them into an optimized artifact ([dspy.ai](https://dspy.ai/)). The "authored modules → compile step → single optimized output" shape mirrors CCS minis→BUNDLE, but DSPy compiles *prompts programmatically against data*, whereas CCS's compile is a deterministic concatenation preserving boundaries. DSPy validates the compile-step intuition without the human-readable-module goal.

**RAG chunk-and-retrieve.** The **parent-document retrieval** pattern — "index small child chunks, return the larger parent" ([SurePrompts](https://sureprompts.com/blog/chunking-strategies-for-rag); [Weaviate](https://weaviate.io/blog/chunking-strategies-for-rag)) — is the retrieval analog of CCS's index-of-minis→load-full-mini. **Difference:** RAG selects by embedding similarity at runtime (lossy, non-deterministic); CCS selects by an authored router and explicit coverage rule (deterministic, auditable). RAG lacks authored modularity, a core, and a compiled bundle.

**Non-AI analogs.** **DITA** is the deepest match: a *topic* = "titled unit understood in isolation, authored self-contained" (= mini); a *ditamap* = "pointers to topics, stores little content" (= INDEX); DITA then *compiles* topics into PDF/HTML/portal deliverables (= BUNDLE), with conditional-processing profiles (= presets) ([OASIS](https://docs.oasis-open.org/dita/dita/v1.3/os/part1-base/archSpec/base/topicdefined.html); [Wikipedia](https://en.wikipedia.org/wiki/Darwin_Information_Typing_Architecture)). DITA has *everything CCS has* plus content-reference reuse, conditional profiling, and a 20-year toolchain — but LACKS the runtime coverage/loading rule (its output is static). **Microkernel/plug-in architecture** maps core→00-core and plug-ins→minis, coordinated by a compact kernel ([O'Reilly](https://www.oreilly.com/library/view/software-architecture-patterns/9781098134280/ch04.html)); it contributes the "stable small core + independent extensions" prior. **Software linkers/tree-shaking** are the direct metaphor for BUNDLE + presets (dead-mini elimination), which no AI framework currently implements.

**Verdict:** No single system combines all five CCS elements. Anthropic Skills and Cursor cover minis+disclosure+core-ish+trigger-hints; llms.txt and DITA cover index+bundle; DSPy/linkers cover compilation. **CCS's distinctive combination is (index + minis + core) *plus a compiled bundle with presets* *plus an explicit runtime coverage threshold*** — the compile step and the "≥60% → load bundle, else minis, else fan-out" policy are what no mainstream analog ships.

## 2. Gaps CCS lacks that mature analogs have

**High-leverage (worth adopting):**
- **Trigger/auto-attach metadata.** Cursor globs+`alwaysApply` and Copilot `applyTo` turn "load when" from prose hints into machine-enforced attachment. CCS's INDEX hints are model-interpreted only; adding glob/description trigger fields would make loading deterministic ([Cursor](https://cursor.com/docs/rules)).
- **Versioning + dependency declarations.** MCP prompts version centrally; plugin marketplaces pin versions. There is an open, unresolved demand for *inter-module dependencies* — Claude Code issue #27113 asks for `package.json`-style declarative skill dependencies, and practitioners report "skills with tight dependencies create fragile systems" ([issue #27113](https://github.com/anthropics/claude-code/issues/27113); [best practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)). CCS should declare mini version + optional cross-mini `requires`, and the compiler should resolve/order them (analogous to a linker).
- **Eval/test harness.** Anthropic's own guidance is "**build evaluations first** … create three scenarios, establish a baseline, then write minimal instructions," and ships an eval JSON schema ([best practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)). CCS has no per-mini eval harness; this is the single highest-leverage add, because §3 evidence shows unmeasured instructions frequently *hurt*.
- **Distribution/registry + caching.** Marketplaces provide discovery, auto-update, and prompt-caching-friendly stable ordering. A stable BUNDLE is inherently cache-friendly (prefix caching) — CCS could lean into this deliberately.

**Over-engineering risks (mature ecosystems built and regretted):**
- **Speculative comprehensiveness.** llms.txt shows the failure mode of "publish a big compiled file nobody consumes": Google stated June 2026 it has *no* effect on rankings/AI Overviews and no major LLM crawler fetches it; adoption stalled outside dev-tooling ([Google/DigitalApplied](https://www.digitalapplied.com/blog/google-llms-txt-no-seo-value-lighthouse-audit-2026); [State of llms.txt 2026](https://presenc.ai/research/state-of-llms-txt-2026)). Lesson: the BUNDLE only pays off if the runtime actually loads it — don't build presets for hypothetical consumers.
- **Deep reference nesting / heavy dependency graphs.** Anthropic explicitly warns to keep references "one level deep" because the model partially reads nested files; and dependency-hell reports counsel "design for independence first" ([best practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)). CCS's mini-to-mini composition should stay shallow; a full dependency DAG is likely over-engineering.
- **Conditional-profiling sprawl (DITA lesson).** DITA's conditional-processing power is widely cited as its adoption barrier (toolchain complexity). CCS presets should stay few and named, not a combinatorial filtering system.
- **Usage telemetry to auto-tune indexes** is attractive but unproven at this granularity; treat as future work, not core.

## 3. Scope: what benefits from modular packaging

Evidence converges on a clear split: **procedural/workflow + non-inferable reference benefit; generic/inferable knowledge does not and can hurt.**

- **Procedural > passive knowledge.** The Agent Skills survey argues skills' advantage over RAG is that "retrieved passages are passive — they cannot prescribe multi-step workflows"; skills "encode procedural knowledge: SOPs, domain conventions, task-specific heuristics" and are modular/portable ([arXiv 2602.12430](https://arxiv.org/html/2602.12430v3); [Ylang Labs](https://ylanglabs.com/blogs/agent-skills)). This directly supports CCS packaging *workflows* as minis over packaging generic facts.
- **Inferable instructions can hurt — the strongest empirical signal.** The ETH Zurich study "On the Impact of AGENTS.md" finds LLM-generated context files *slightly reduce* success (~0.5pp SWE-bench Lite; 2–3pp AgentBench) while agents "spend 14–22% more reasoning tokens and take 2–4 extra steps"; human-written files gave only ~4% AgentBench gain, and the authors recommend "omitting LLM-generated context files entirely and limiting human-written instructions to non-inferable details (specific tooling, custom build commands)" ([arXiv 2601.20404](https://arxiv.org/html/2601.20404v2); [InfoQ](https://www.infoq.com/news/2026/03/agents-context-file-value-review/); [heise](https://www.heise.de/en/background/AGENTS-md-Helpful-agent-briefing-or-token-hog-11245317.html)). A companion study ("Evaluating AGENTS.md," [arXiv 2602.11988](https://arxiv.org/pdf/2602.11988)) reports instructions are well-followed yet don't significantly improve outcomes. **CCS implication:** minis should carry *non-inferable* domain specifics and traps (exactly what 00-core targets), not restate what the model already knows — validating CCS's "concise, only what Claude lacks" premise and making the coverage rule (don't load irrelevant minis) load-bearing for avoiding the token/step penalty.
- **Benchmarks now exist.** SkillsBench benchmarks how well agent skills work across diverse tasks ([arXiv 2602.12670](https://arxiv.org/pdf/2602.12670)); CTXBENCH/SWE-Context-Bench measure whether actively-used context files help solve real SWE tasks and reuse prior context ([SWE Context Bench](https://arxiv.org/pdf/2602.08316)). These are the natural venues to evaluate CCS's bundle-vs-minis-vs-fanout policy empirically.
- **Tool/API specs and style guides:** MCP treats API/tool specs as separate resources/tools, not skill prose, and Anthropic recommends deterministic *scripts* over instructions for fragile operations ("low degrees of freedom") — suggesting API/tooling belongs in executable Level-3 resources, while style guides (low variability, always-relevant) fit an always-loaded core rather than a triggered mini ([best practices](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/best-practices)).

**Net:** modular packaging pays off most for *procedural workflows and non-obvious domain specifics*; it is neutral-to-negative for generic reference the model can infer — which is precisely the case for CCS's conciseness discipline, its coverage rule, and a per-mini eval harness as the top gap to close.
